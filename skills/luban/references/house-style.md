# LearnPrompt House README 模板

> 鲁班"亮活"步骤的标配模板。新仓库出生即用,旧仓库对齐时逐节替换。
> 风格基因:**模块化分节 + 人感开场 + 真实数字 + 零API底色**。

## 风格铁律

从万星级 README(huashu-design、zhangxuefeng-skill、blogger-distiller)和自家实战(ai-news-radar、afu、luban)提炼,共十条:

1. **钩子是一句引语,不是功能清单。** 标题下用 `> *「一句让人停下来的话」*`——讲它替用户省掉什么痛苦,或者最有性格的一句话。
2. **首屏 10 秒讲清价值。** hero 区只放:名字、引语、徽章、一行加粗的价值陈述、锚点导航。其余全部下沉。
3. **人感开场。** 正文第一节像跟朋友说话("事情是这样的""你研究了 10 个对标博主,你能说清楚任何一个吗?"),用第二人称戳痛点,不用产品文档腔。
4. **产物前置。** GIF/截图/真实输出样例放在安装命令之前——先让人想要,再告诉怎么装。效果示例用真实运行产物,禁止虚构样例。
5. **一行安装,30秒上手。** `npx skills add owner/repo` 一行;装完"第一句话"直接给出可复制的 prompt。
6. **每个数字可点击查证。** 战绩、星数、回放条数,全部挂 PR/release/数据文件链接。给不出链接的数字不写。
7. **模块化分节。** 每节只干一件事,节与节之间用 `---` 分隔,首屏锚点导航串起全文。读者跳着读不迷路。
8. **不写大词。** "支持一切""全自动""彻底解决"禁用;对比同行用表格、不攻击。
9. **零API底色。** 能零Key就零Key、fork即拥有;做不到就在显眼处写清成本和前置条件(参考 blogger-distiller 的成本估算节)。
10. **双语策略。** 中文为主 `README.md`,有外部受众时配 `README.en.md`,首行互链:`<sub>🌐 <b>中文</b> · <a href="README.en.md">English</a></sub>`。

## 家族署名条的排版规矩

署名条(同门互链 footer)统一从母版复制,不手写。排版三条死规矩,来自一次真实翻车(`<sub>` 跨段包裹导致 GitHub 渲染换行重叠):

1. **行内标签不准跨段。** `<sub>`/`<b>` 这类行内 HTML 只许包单行内容;多段内容包进一个 `<sub>` 会被渲染器强行重排,行高错乱、文字叠字。
2. **链接链写在同一物理行。** 同门链接全部放一个 markdown 段落、一行源码,靠自然换行折行;不要一行一个链接尾随 `·` 拼段落。
3. **改署名条 = 改全家。** 母版改了就要铺到所有仓库,发布后用 `gh api repos/<owner>/<repo>/readme -H "Accept: application/vnd.github.html"` 验一次渲染产物(数链接数、查 `<sub>` 配平),不要只看源码。

## 对齐旧仓库时的最小动作序

不要推倒重写。按这个顺序逐项替换,每项一个提交:

1. hero 区(引语钩子 + 徽章 + 一行价值 + 锚点导航)
2. 产物前置(把 GIF/截图挪到安装之前;没有就先录,vhs tape 入库)
3. 一行安装 + 装完第一句话
4. 真实数字挂链接
5. 补缺失的节(对比/安全边界/验证)
