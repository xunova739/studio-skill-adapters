# Excalidraw 图表生成器

## 流程入口

**第零步（每次触发都执行，静默）：**
```bash
# 埋点
python ~/.claude/skills/excalidraw-diagram-skill/scripts/ping_usage.py "excalidraw-diagram" "<图表类型>" &

# 自动安装验证 Hook（幂等，已安装则跳过）
bash ~/.claude/skills/excalidraw-diagram-skill/hooks/install-hooks.sh 2>/dev/null
```

**第一件事：判断有没有内容来源。**

| 情况 | 下一步 |
|------|------|
| 用户提供了钉钉文档/表格链接 | → 用本地脚本 `dingtalk/scripts/` 拉取内容，进入**确认环节** |
| 用户提供了本地文件或直接粘贴了内容 | → 读取内容，进入**确认环节** |
| 用户只说了一个想法，没有任何文档 | → 进入**需求对话** |

**钉钉内容拉取命令：**

```bash
# 钉钉文档
SKILL_DIR=$(dirname $(realpath ~/.claude/skills/excalidraw-diagram-skill/SKILL.md 2>/dev/null || echo ~/.claude/skills/excalidraw-diagram-skill/SKILL.md))
python "$SKILL_DIR/dingtalk/scripts/dingtalk_doc.py" "<钉钉文档URL>"

# 钉钉电子表格
python "$SKILL_DIR/dingtalk/scripts/dingtalk_sheet.py" "<表格URL>" "<Sheet名称>"

# 钉钉 AI 表格
python "$SKILL_DIR/dingtalk/scripts/dingtalk_ai_table.py" "<AI表格URL>"
```

> **首次使用**：将 `dingtalk/config/servers.json.example` 复制为 `dingtalk/config/servers.json`，填入钉钉 MCP 凭据。

## 需求对话（无文档时）

目标：用最少的问题，拿到画图所需的 4 项信息。拿到就开始，不要继续追问。

**完成标准（拿到这 4 项即可）：**
1. **目的**：这张图是为了说明什么？给谁看？
2. **内容骨架**：有几个核心概念/步骤/角色？
3. **图表类型**：流程图、架构图、思维导图、对比图，还是其他？
4. **风格**：精准线条（汇报/技术文档用）还是手绘感（头脑风暴/草稿用）？

**追问规则（借鉴 demand-discovery）：**
- 一轮只问一个问题，问当前最缺的那项
- 每个问题都先给推荐答案，用户只需要说"对"或"改成X"
- 能从上下文推断的就推断，不问已知信息

**推荐格式：**
```
我的理解是：你想画一张 [图表类型]，用来说明 [目的]。

我的推荐：[具体建议]

你只需要回答：同意，还是改成 X？
```

## 文档理解（有文档时，必须先做这一步）

拿到文档或表格内容后，**不要直接开始画图**。先完成以下分析，再输出给用户确认：

1. **主题识别**：这份文档在讲什么？核心问题/目的是什么？
2. **结构提炼**：有哪些主要实体、步骤、角色或数据维度？它们之间是什么关系（顺序、层级、并列、依赖）？
3. **内容分类**：哪些内容适合作为图的主干？哪些是补充说明？哪些可以作为证据内容（数据样例、具体数值、真实事件名）放进图里？
4. **图表类型推荐**：根据结构特征推荐最合适的类型，并说明理由

**输出给用户确认的格式：**
```
我读了这份文档，核心是：[1-2句话说清楚文档在讲什么]

主要结构：
- [实体/步骤1] → [实体/步骤2] → ...
- [并列或层级关系说明]

我推荐画一张 [图表类型]，因为 [理由]。
风格建议：[精准线条/手绘感]，因为 [判断依据]。

这个方向对吗？有什么要调整的？
```

用户确认后进入**图表设计**。

## 风格映射

风格不写死，根据场景推断。用户可随时覆盖。

| 场景信号 | roughness | 说明 |
|----------|-----------|------|
| 业务汇报、技术架构、对外展示 | `0` | 精准线条，干净专业 |
| 头脑风暴、草图、内部讨论 | `1` | 轻微手绘感 |
| 用户说"手绘"、"草稿"、"随意" | `2` | 明显手绘风格 |
| 钉钉表格数据驱动 | `0` | 数据图要清晰 |

`fontFamily` 固定用 `3`（简洁无衬线）。`opacity` 固定 `100`，不用透明度做层次。

## 图表设计规则

### 核心原则：图表要论证，不只是展示

**同构测试**：拿掉所有文字后，结构本身还能表达概念吗？不能就重新设计。

**容器克制**：默认用自由浮动文字，只在有意义时加形状。目标：< 30% 的文字元素在容器内。

### 形状语义

| 概念类型 | 形状 | 理由 |
|----------|------|------|
| 标签、描述、注释 | 无（自由浮动文字） | 排版本身就是层级 |
| 时间轴节点、要点标记 | 小椭圆（10-20px） | 视觉锚点，不是容器 |
| 开始/触发/输入 | `ellipse` | 柔和，起点感 |
| 结束/输出/结果 | `ellipse` | 完成感 |
| 决策/条件 | `diamond` | 经典决策符号 |
| 流程步骤/动作 | `rectangle` | 有边界的动作 |
| 数据库/存储 | 圆柱体（从图标模板库 `database` 取） | 经典数据库符号，一眼可识别。仅用于架构图 |
| 抽象状态/上下文 | 重叠椭圆 | 模糊、云状 |

### 视觉模式库

| 概念行为 | 用这个模式 |
|----------|-----------|
| 一对多输出 | 扇出（中心 → 多个分支） |
| 多对一汇聚 | 汇聚（漏斗，多箭头合一） |
| 层级/嵌套 | 树形（线 + 自由浮动文字，不用方框） |
| 循环迭代 | 螺旋/环形（箭头回到起点） |
| 输入→处理→输出 | 流水线（前 → 中间 → 后，展示前后对比） |
| 阶段切换 | 间隔/分隔线（视觉断开） |
| 多角色协同 | 泳道图（横向角色泳道 × 纵向阶段列，见下方泳道规则） |
| 数据存储/数据库 | 圆柱体图标（见形状语义表），一个就够，不要画多个 |
| 外部系统/API | 云朵形状或带波浪边的矩形 |
| 用户/角色 | 简笔人形 + 角色名文字 |

### 图表思考激发

**不要只画"方块+箭头"。** 好的图表用视觉隐喻让概念一眼可懂。在设计图表结构时，先问自己：

1. **这个概念像什么？** 漏斗、管道、洋葱层、冰山、天平、齿轮组……找到一个日常物体的隐喻，比抽象方块更容易记住
2. **读者最关心什么？** 是整体结构？是某个关键节点？是数据流向？最关心的部分要用最大、最醒目的视觉元素
3. **有没有对比关系？** 前后对比、好坏对比、理想 vs 现实——对比用并排布局 + 颜色反差
4. **有没有层次关系？** 从外到内（同心圆/洋葱图）、从上到下（金字塔）、从粗到细（漏斗）

**常用视觉隐喻：**

| 概念 | 视觉隐喻 | 画法 |
|------|----------|------|
| 筛选/转化 | 漏斗 | 上宽下窄的梯形，每层标注转化率 |
| 防护/安全 | 盾牌 | 盾形轮廓内放安全策略 |
| 核心 vs 外围 | 同心圆 | 最重要的在中心，依赖层向外扩展 |
| 优先级 | 金字塔 | 顶部最重要，底部最基础 |
| 时间演进 | 时间轴 | 横向或纵向线 + 节点标注时间点 |
| 并行处理 | 泳道/管道 | 多条平行通道同时推进 |
| 输入输出 | 管道 | 左入右出，中间是处理逻辑 |

### 图标模板库（仅限架构图/技术图）

`libraries/icon-templates.json` 里预存了 6 个从 Excalidraw Libraries（MIT 协议）策展的图标模板：

| 图标 | 关键字 | 用途 |
|------|-------|------|
| `database` | 数据库、存储 | 圆柱体形状，蓝色 |
| `server` | 服务器、微服务、后端 | 机架形状 |
| `cloud` | 云服务、外部 API、第三方 | 云朵形状 |
| `cache` | 缓存、Redis、CDN | 扁圆柱 |
| `queue` | 消息队列、事件总线 | 菱形组合 |
| `document` | 文档、配置、日志 | 折角纸张 |

**使用规则——图标不是装饰，是符号：**

- **架构图 / 系统设计图 / 技术方案**：可以使用图标模板。这些图标是行业通用符号（数据库圆柱、云朵等），读者一眼就能识别组件类型
- **流程图 / SOP / 泳道图 / 工作流**：**不使用图标**。流程图的读者关心"我该做什么"，图标是干扰。用矩形 + 文字就够
- **汇报演示**：克制使用，同类元素不要超过 2 种图标

**使用方法：** 读取 `libraries/icon-templates.json`，取出对应模板的 `elements` 数组，调整 `x`/`y` 坐标放到目标位置，给每个元素重新分配唯一 `id`，保持 `groupIds` 一致使图标作为整体移动。

### 布局规则库（所有图类型通用）

**画任何图之前，必须先读 `references/layout-rules.md`**。

布局规则库定义了每种图类型的：
- **可重复单元**：每种图的最小构建块（节点块的形状、尺寸、间距）
- **组装规则**：单元怎么排列（横向/纵向/放射），坐标自动计算
- **箭头语义**：不同类型的箭头用不同线型和颜色区分

核心机制：AI 只提取节点列表和关系，然后按规则重复单元 × N 次，自动算坐标。不靠 AI 发挥布局。

### 模板脚本（有模板的图类型必须先用模板生成骨架）

三种高频图类型已有模板脚本，**必须先用模板生成骨架，再在骨架上填充内容**：

| 图类型 | 模板脚本 | 用法 |
|--------|---------|------|
| 泳道图 | `libraries/swimlane-template.py` | `python swimlane-template.py --phases 5 --roles 3` |
| 流程图 | `libraries/flowchart-template.py` | `python flowchart-template.py --steps 7 --title "审批流程"` |
| 架构图 | `libraries/architecture-template.py` | `python architecture-template.py --config config.json` |

三个模板都支持 `--config config.json` 做完全自定义（节点名称、颜色、数量），也支持命令行参数快速生成默认骨架。

### 泳道图（Swimlane）

适用场景：多角色协同流程、SOP、跨部门工作流。

**画泳道图时，必须先读 `libraries/swimlane-guide.md`**，里面包含模板用法、布局规则、配色方案、箭头铁律。读完再动手。

### 多层次架构（复杂图）

1. **Level 1 概览**：顶部或底部一行简化流程，给整体感
2. **Level 2 区块**：用背景色或分隔线划分功能区
3. **Level 3 细节**：区块内的代码片段、实际数据、真实事件名

技术图必须有 Level 3 的证据内容（真实 API 名、真实字段、真实示例），不能只有标签框。

### 正反例

**❌ 展示型（避免）**
```
[输入] → [处理] → [输出]
```
五个等大方框，每个方框里一个词，没有任何证据说明"处理"到底做了什么。

**✅ 论证型（目标）**
```
简历 PDF
  ↓
[AI 筛选]  ←── 六维评分模型（文字说明评分维度）
  ├── 通过 → [业务审简历]（展示实际审核检查项）
  └── 拒绝 → 钉钉通知申请人（展示通知模板内容）
```
每个节点展示"实际长什么样"，不只是叫什么名字。

### 布局原则

- **留白 = 重要性**：最重要的元素周围留充裕空间，次要元素适当紧凑
- **流向**：序列用左→右或上→下，辐射用中心放射
- **连接必须明确**：位置相近不等于有关系，有关系必须画箭头
- **层级通过大小**：主角最大，逐级缩小，至少保持 3 级大小差异

### 视觉原则（什么是"好看"）

以下原则基于格式塔心理学和信息可视化研究，描述人眼感知的规律。不是硬编码参数——具体数值由图的尺度和密度决定，但原则不变。

#### 1. 可辨性（Discriminability）

**人眼无法分辨互相遮挡的信息。** 任何两个独立元素——文字、形状、线条——在视觉上必须能被单独识别，不能有重叠或模糊边界。

- **文字间距**：两段文字之间的空白至少等于较小字号的行高。如果缩放后两段文字在视觉上贴在一起，说明间距不够
- **形状间距**：两个形状之间的空白至少等于较小形状边长的 1/3。紧贴的形状会被感知为一个整体
- **线条与元素**：线条不得穿过不相关的形状或文字。如果路径上有障碍物，必须绕行

**自检方法**：把图缩小到 50%，如果任何两个元素看起来"粘在一起"，就需要拉开距离。

#### 2. 路径清晰性（Path Clarity）

**箭头是读者的视觉导轨，不能让导轨穿墙。** 当箭头需要跨越其他元素时，必须用折线绕行，而不是直接穿过。

- **折线路由**：跨区域的箭头使用 L 形或 Z 形折线，沿空白通道走。折点放在元素之间的间隙中
- **最小交叉**：当多条箭头交叉不可避免时，用虚线 vs 实线区分，或用颜色区分语义
- **箭头标签**：标签放在箭头路径的空白处（通常在折点旁），不能压在其他元素上
- **方向一致性**：同一层级的箭头尽量保持统一方向（都向右、都向下），避免来回穿梭

**原理**：人眼会沿箭头方向追踪路径。如果箭头穿过一个方块，读者会误以为方块是路径的一部分。

#### 3. 层次感（Visual Hierarchy through Depth）

**平面图需要"伪深度"来区分前景和背景。** 通过颜色饱和度和透明度的差异，让不同层级的元素在视觉上分离。

- **背景层**：用极低饱和度、高透明度的色块标识区域归属（如泳道背景、功能分区）。背景不应抢夺前景注意力
- **中间层**：活动节点、流程步骤等核心内容，用饱和度适中的填充色 + 清晰边框
- **前景层**：当前焦点、关键路径、异常状态，用高饱和度或高对比色突出
- **分隔线**：区域之间的分隔用极浅色或虚线，只起"暗示边界"的作用，不能比内容元素更醒目

**原理**：人眼优先关注高对比度元素。如果背景色太重，会和前景竞争注意力，导致"图很花但看不清重点"。

#### 4. 呼吸感（Whitespace as Structure）

**留白不是浪费空间，是结构的一部分。** 相关元素靠近，不相关元素远离——间距本身在传递信息。

- **组内紧凑**：属于同一组的元素（如同一泳道内的步骤）间距均匀且较小
- **组间疏朗**：不同组之间（如不同泳道、不同阶段）间距明显大于组内间距，至少 2 倍以上
- **边距充足**：图的四边留足空白，不要让内容贴到画布边缘。边距至少等于最大元素尺寸的一半

**原理**：格式塔接近性原则——距离近的元素被感知为一组。如果所有间距一样大，读者无法区分哪些元素属于同一组。

#### 5. 视觉重心（Balance）

**图不能"倒向一边"。** 元素的视觉重量（大小 × 颜色深浅）在画面上应大致均衡。

- 如果右侧堆了很多元素，左侧空了一大片，整张图会让人感觉不稳定
- 解决方案：调整间距重新分配空间，或给空旷区域添加注释/说明文字作为视觉锚点
- 对称不等于均衡——非对称布局只要视觉重量均匀分布同样稳定

#### 6. 连接跟随性（Bound Connections）

**箭头必须"粘"在元素上，不能只是视觉对齐。** 用户拖动方块时，箭头应跟着移动，而不是留在原地断开。

- **Excalidraw 实现**：每个箭头的 `startBinding` 和 `endBinding` 必须指向目标元素的 `id`，同时目标元素的 `boundElements` 数组必须包含该箭头的 `id`。这是双向绑定——缺任何一侧都会导致断开
- **检查方法**：每条箭头都检查：`startBinding.elementId` 对应的元素是否存在？该元素的 `boundElements` 里有没有这条箭头？`endBinding` 同理
- **自由浮动文字例外**：标注性文字（如补充说明、备注）不需要绑定，它们本来就是独立于流程的

**原理**：图表是动态文档，用户会反复调整布局。如果箭头不跟随，每次调整都要手动重新连线，体验极差。

## 配色

**所有颜色从 `references/color-palette.md` 取，不自行发明颜色。**

用户可编辑 `color-palette.md` 换成自己品牌色，其余规则不变。

颜色编码意义，不编码装饰：

| 语义 | 用途 |
|------|------|
| Primary/蓝 | 默认流程节点 |
| Start/橙 | 触发点、入口 |
| End/绿 | 成功结果、输出 |
| AI/紫 | AI 处理节点 |
| Error/红 | 错误、拒绝、警告 |
| Decision/黄 | 判断节点 |

文字层级靠颜色区分，不靠加框：标题 `#1e40af`，副标题 `#3b82f6`，说明文字 `#64748b`。

## 大图分段生成策略

**复杂图必须分段写 JSON，不能一次生成全部。**

原因：单次输出 token 上限 ~32k，复杂图轻易超标；分段质量更高，每段可单独检查。

流程：
1. 建基础文件（JSON 结构 + 第一个区块）
2. 每次追加一个区块，用描述性 ID（如 `"trigger_rect"`，不用随机字符串）
3. 区块 ID 按前缀区分段落（第一段 `100xxx`，第二段 `200xxx`）
4. 跨段箭头：写新段时同步更新被连接元素的 `boundElements`
5. 全部写完后整体检查一遍绑定关系

## 渲染验证（必须执行，不可跳过）

**JSON 不能替代眼睛。** 生成 JSON 只是一半，另一半是看图、发现问题、修改、再看。这个循环是 skill 质量的核心，不是可选步骤。

### 渲染命令

```bash
cd ~/.claude/skills/excalidraw-diagram-skill/references
uv run python render_excalidraw.py <文件路径>
```

输出 PNG 到同目录，然后用 **Read 工具**打开 PNG 看图。

### 几何验证命令（渲染前或渲染后必跑）

```bash
cd ~/.claude/skills/excalidraw-diagram-skill/references
python3 validate_geometry.py <文件路径>
```

检查 4 项：
1. **折线路径**：elbowed 箭头的每段是否严格水平/垂直（不允许斜线段）
2. **文字遮挡**：容器内文字是否被更高层级的元素覆盖
3. **元素重叠**：非容器元素之间是否有意外重叠导致不可读
4. **箭头绑定**：startBinding/endBinding 与目标元素的 boundElements 是否双向一致

**必须在最终交付前跑一次，所有检查通过才算完成。**

首次使用需安装：
```bash
cd ~/.claude/skills/excalidraw-diagram-skill/references
uv sync
uv run playwright install chromium
```

### 验证循环（每次生成后必须跑）

**第 1 步：渲染 + 看图**
运行渲染脚本，用 Read 工具打开 PNG。

**第 2 步：对照原始设计意图审查**

先不找 bug，先问：这张图有没有表达出你设计的东西？

- 视觉结构有没有反映你选的模式（扇出/汇聚/树/环/流水线）？
- 最重要的元素在视觉上最突出吗？
- 眼睛知道从哪里开始看、按什么顺序看吗？
- 技术图里有没有证据内容（真实数据/API 名/字段）？

**第 3 步：检查视觉缺陷**

逐条对照"视觉原则"检查：

| 视觉原则 | 检查要点 |
|----------|----------|
| 可辨性 | 有没有文字/形状重叠？缩小到 50% 还能分辨吗？ |
| 路径清晰性 | 箭头有没有穿过不相关的元素？跨区域箭头是否用了折线绕行？ |
| 层次感 | 背景区域颜色是否够淡、不抢前景？关键节点是否视觉突出？ |
| 呼吸感 | 组内间距小于组间间距吗？边距是否充足？ |
| 视觉重心 | 图有没有"倒向一边"？元素分布是否大致均衡？ |

另外检查这些技术缺陷：

| 缺陷类型 | 具体表现 |
|----------|----------|
| 文字溢出 | 文字被容器裁切，或超出容器边界 |
| 箭头悬空 | 箭头指向错误元素或悬空没有终点 |
| 文字过小 | 渲染后字号太小，无法阅读 |
| 标签歧义 | 文字标签不清楚属于哪个元素 |

**第 4 步：修改 JSON**

根据所见问题修改坐标、尺寸、文字、箭头路径：
- 文字溢出 → 加宽容器 `width`
- 元素重叠 → 调整 `x`/`y` 拉开距离
- 箭头穿越 → 给 `points` 数组加折点绕开障碍
- 标签歧义 → 移动标签靠近所描述的元素
- 构图失衡 → 整体平移某个区块，重新分配空间

**第 5 步：再渲染、再看图**

**第 6 步：重复，直到满足停止条件**

---

### 何时停止

满足以下所有条件才可以停：

- [ ] 渲染结果和你设计时的概念匹配
- [ ] 没有文字溢出、元素重叠
- [ ] 箭头连接到正确目标，没有穿越其他元素
- [ ] 间距均匀，构图不失重
- [ ] 文字在渲染尺寸下可读
- [ ] 你愿意直接把这张图发给用户

**通常需要 2-4 轮。** 不要因为"没有明显 bug"就停——构图可以更好就要继续改。

## 版本备份（防覆盖机制）

Devix Excalidraw 插件已内置外部变更检测和版本备份：
- 插件每 2 秒检测文件是否被外部修改，如检测到则自动重新加载
- 插件每次保存时自动备份旧版本到 `.versions/` 目录（间隔 > 30 秒才备份，最多 20 个）
- 插件顶部有「历史记录」按钮，用户可以浏览和恢复历史版本

### Skill 端规则

尽管插件已有自动备份，Skill **修改已有文件前仍必须主动快照**（插件的自动备份依赖保存间隔，可能跳过关键版本）：

1. **修改前必须快照：** 用 Python 将当前文件复制到 `.versions/{文件名}_{YYYYMMDD}T{HHMMSS}.json`
2. **新建文件不需要快照**（没有旧版本可保护）
3. **批量修改只需一次快照**（修改前快照一次即可，不需要每改一个属性都快照）

### 快照代码模板

```python
import shutil, os, time
filepath = '/home/admin/workspace/excalidraw/项目名/文件.excalidraw.json'
versions_dir = os.path.join(os.path.dirname(filepath), '.versions')
os.makedirs(versions_dir, exist_ok=True)
base = os.path.basename(filepath).replace('.excalidraw.json', '')
ts = time.strftime('%Y%m%dT%H%M%S')
shutil.copy2(filepath, os.path.join(versions_dir, f'{base}_{ts}.json'))
```

### 版本恢复

用户说"回退"时，两种方式：
- **推荐**：让用户在插件里点「历史记录」按钮自助恢复
- **备选**：从 `.versions/` 目录找到对应版本，复制覆盖回主文件（覆盖前同样先快照当前版本）

## 反馈沉淀机制

**Skill 应该越用越准。** 每次用户对图表不满意、主动修改、或要求返工，都是沉淀的机会。

### 触发条件

以下任何一种情况出现时，执行沉淀流程：
- 用户说"不对"、"改一下"、"不是我想要的"
- 用户自己编辑了图表后告诉你哪里不好
- 渲染后发现视觉缺陷（哪怕用户没说）
- 用户对同一类图反复提出相似的修改意见

### 沉淀流程（每次出错都跑）

```
1. 定性：这是哪类错误？
   [布局] / [语义] / [密度] / [结构] / [JSON] / [流程]

2. 归因：为什么会犯这个错？
   - 是选错了视觉模式？
   - 是信息量判断失误？
   - 是对用户意图理解偏了？
   - 是技术原因（坐标、绑定）？

3. 判断可抽象性：这个错误只属于这次，还是以后也可能犯？
   └─ 只属于这次 → 修了就好，不沉淀
   └─ 可以抽象 → 进入第 4 步

4. 判断沉淀类型：
   └─ 是可靠的失败模式 → 加入 references/prohibition-list.md（禁止清单）
   └─ 是发现了好的做法 → 加入 references/positive-patterns.md（正例库）
   └─ 是一个典型场景 → 加入 references/test-cases.md（测试用例）

5. 写入对应文件，并更新触发计数：
   - 新增条目：触发次数设为 1，最后触发设为今天
   - 已有条目被再次触发：触发次数 +1，更新最后触发日期

6. 定期维护（每累计使用 10 次扫描一次）：
   - 触发次数 0 且超过 30 天 → 移入"候选清理"区
   - 两条规则来源场景高度重叠 → 标记"候选合并"，保留更抽象的那条
   - 有测试用例完全覆盖的规则 → 标记"候选毕业"，从显式列表移入已毕业区
```

### 禁止清单条目格式

每条禁止规则必须包含：

```
❌ [分类] 规则描述（一句话，可执行）
   为什么：不遵守会导致什么问题
   替代方案：应该怎么做
   来源场景：哪类需求触发了这条规则（抽象描述，不写具体用户名）
```

**注意**：禁止清单不是越长越好。每条加入前都要问：这条规则能防止真实的失败吗？如果不能，不加。

禁止清单存放在 `references/prohibition-list.md`。**当条目超过 20 条时，按错误分类拆分为独立文件**（如 `references/prohibitions/layout.md`、`references/prohibitions/json.md`），SKILL.md 中只保留每类最高频的 2-3 条，其余按需读取。

### 正例模式条目格式

```
✅ [分类] 模式描述
   适用场景：什么情况下用这个模式
   为什么有效：解决了什么问题
   来源：从哪类场景归纳出来的
```

### 每次修改后的自检问题

修完图、用户确认满意后，对自己问：
1. 这次犯的错，下次还会犯吗？
2. 有没有可以抽象成规则的东西？
3. 需要加禁止清单吗？还是加正例？还是加测试用例？

## 质量自检清单

生成完、渲染完，对照检查：

### 内容质量
- [ ] 技术图有证据内容（真实 API 名/字段/数据样例），不只是标签
- [ ] 拿掉文字后结构本身还能表意
- [ ] 每个主要概念用了不同视觉模式，没有统一卡片网格

### 容器克制
- [ ] < 30% 文字元素在形状容器内
- [ ] 树形/时间轴用线 + 自由文字，没有用方框
- [ ] 字号和颜色已经创造了层级，不需要再加框

### 结构正确
- [ ] 每个关系都有箭头或线连接
- [ ] 视觉流向清晰，眼睛知道从哪里开始看

### JSON 规范
- [ ] `text` 字段只有可读文字，无转义或代码
- [ ] `fontFamily: 3`
- [ ] `opacity: 100`（所有元素）
- [ ] `roughness` 根据风格场景设置，非固定值

### 渲染检查
- [ ] 已渲染为 PNG 并用 Read 工具查看
- [ ] 无文字溢出、元素重叠
- [ ] 箭头连接到正确目标
- [ ] 间距均衡，构图不失重
