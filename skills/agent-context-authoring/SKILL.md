# Agent Context Authoring

目标：写出短、稳、能约束 Agent 行为的常驻上下文文件，而不是项目百科。

本 Skill 用于三类文件：

- `AGENTS.md`：项目或目录级常驻入口。
- `CLAUDE.md`：Claude Code 特有差异层。
- `.claude/agents/*.md` 或类似角色文件：某个子代理/角色的按需说明。

## 核心判断

常驻上下文每次都会被模型读取。它只应该写：

- Agent 每次都要遵守的行为红线。
- 代码、测试、配置看不出来的隐性约定。
- 会稳定影响执行质量的边界、路由和完成标准。

不要写：

- 项目百科、目录树、文件路径清单、函数名清单。
- 框架默认行为、代码一读就知道的事实。
- 长篇背景、教程、案例、复盘、排错手册。

长流程放 Skill / Wiki / README / ARCHITECTURE / INTERFACES；常驻入口只保留路由。

## 两套来源的分工

Karpathy 行为护栏解决 Agent 的执行偏差：

- 不要默默假设；有歧义就说出假设和取舍。
- 简洁优先；不要添加未请求的功能和抽象。
- 外科手术式修改；只碰任务相关内容。
- 目标驱动执行；用可验证标准循环到完成。

AGENTS.md 精简三原则解决常驻上下文污染：

- 控制指令预算：越常驻，越要短。
- 描述能力，不写路径。
- 移除 Agent 可自行发现的信息。

## 执行流程

### 1. 先判定文件类型

写之前先判断目标文件属于哪一层：

| 文件类型 | 作用 | 默认长度 |
|---|---|---|
| `AGENTS.md` | 项目/目录总规则和路由 | 50-120 行 |
| `CLAUDE.md` | Claude Code 差异层 | 20-80 行 |
| 子代理说明 | 单个角色的任务边界、输入、输出 | 40-120 行 |
| Skill `SKILL.md` | 按需流程说明 | 可更长，但要渐进披露 |

如果用户没有明确目标文件，先输出建议结构，不要直接写入常驻文件。

### 2. 收集隐性约定

只收集这些信息：

- 项目定位：一句话说明这个目录是什么。
- 行为红线：哪些动作不能擅自做。
- 权威来源：需求、状态、接口、验证证据以哪里为准。
- 路由规则：什么任务交给哪个 Skill、文档或工作区。
- 验证标准：结束前必须证明什么。

不收集文件树、函数名、实现细节和代码可见事实。

### 3. 生成短版

优先按以下模板写。

`AGENTS.md`：

```markdown
# Project Agent Rules

## 定位
一句话说明这个项目/目录是什么，以及 Agent 在这里负责什么。

## 常驻约定
- 只写代码和目录看不出来、但会影响执行的约定。
- 修改代码时保护用户已有改动，只做任务相关变更。
- 非琐碎任务必须有可验证完成标准。

## 路由
- 资料/需求/代码/验证分别去哪里。
- 长流程走对应 Skill 或文档，不在这里展开。

## 完成标准
- 任务结束前至少要收集哪些证据。
```

`CLAUDE.md`：

```markdown
# Claude Rules

通用项目规则以 AGENTS.md 为准；本文件只写 Claude Code 特有差异。

- 与 AGENTS.md 重复的句子应删除。
- 只记录 Claude 工具、子代理、权限、交互方式的差异。
- 长流程引用 Skill 或文档，不复制到这里。
```

子代理说明：

```markdown
# agent-name

## 使命
一句话说明这个 agent 负责什么。

## 适用场景
- 用户请求什么时使用。
- 哪些任务不要使用。

## 输入
- 需要读取的信息。
- 不需要读取的信息。

## 输出
- 必须交付的结果。
- 必须包含的证据。

## 边界
- 不做哪些事。
- 何时交回主 agent。
```

### 4. 做瘦身审查

生成后逐条检查：

1. 这条是否对大多数任务都生效？
2. 这条是否是代码、测试、配置无法直接推断的隐性约定？
3. 如果它过时，会不会误导 Agent？
4. 它能否缩成一句可执行规则？
5. 它是否应该移到 Skill / Wiki / README / ARCHITECTURE / INTERFACES？

删除或迁移以下内容：

- 具体文件路径和函数名。
- 目录树和路径表。
- 代码可发现的事实。
- 教程式背景和长案例。
- 重复的价值观口号。

如果已安装 `agents-md-slim`，最后调用它或按它的三原则复核；未安装时手动执行同样审查。

## 输出要求

默认输出三段：

1. **建议写入内容**：可直接放进目标文件。
2. **被排除内容**：说明哪些信息不应进入常驻上下文，应放到哪里。
3. **瘦身检查结果**：列出是否还有路径、冗余、代码可发现信息。

如果用户明确要求直接修改文件，才写入文件；否则先给可审阅草案。

## 质量标准

合格的 Agent 常驻上下文应该满足：

- 根入口短，不超过必要范围。
- 规则能指导行为，而不是复述项目事实。
- 没有易过时路径清单。
- 有明确完成标准。
- `CLAUDE.md` 不重复 `AGENTS.md`。
- 子代理说明只写该角色的使命、输入、输出和边界。

一句话记忆：先用 Karpathy 管住 Agent 行为，再用 slim 三原则删掉常驻噪声。
