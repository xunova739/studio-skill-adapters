# Demand Discovery

## 目标

把用户的一句模糊想法聊到足够清楚，最后可以停在三种结果之一：

- **需求雏形**：业务问题、用户、输入输出、边界和下一问已经清楚。
- **SPEC-ready**：可以直接整理成 SPEC / PRD / feature brief。
- **执行-ready**：可以交给 `plan` / `ralplan` / `ralph` / `team` 做计划或实现。

本 Skill 是总调度器，不替代子 Skill。它负责判断当前需求成熟度，选择下一段流程，并保持“先推荐答案，再问一个最高杠杆问题”的节奏。

## 调度顺序

按需推进，不要机械跑完整链路：

1. **想法很早期**：使用 `idea-exploration`。
   - 把口语想法翻译成业务问题、产品能力地图、AI 可行性和 MVP/P0/P1/P2。
   - 适合“我有个想法”“能不能做”“帮我扩一下边界”。
2. **方向已有但假设没压实**：做 `grill-me` 式压力追问。
   - 不要求外部 `grill-me` 已安装；如果没有，就在本 Skill 内完成同等压力测试。
   - 每次只追问一个假设、边界或取舍，并给推荐答案。
3. **重大意图仍不清楚**：使用 `deep-interview`。
   - 适合用户明确说“别假设”“问透一点”“这个会影响后续开发/协作/组织决策”。
   - 必须先问意图、非目标、决策边界，再问实现细节。
4. **需求已稳定但还不能执行**：使用 `plan` 或 `ralplan`。
   - `plan` 用于普通执行计划。
   - `ralplan` 用于需要 Planner / Architect / Critic 共识的复杂或高风险事项。
5. **用户要文档**：使用 `pm-spec` 或本 Skill 的 SPEC 输出模板。
   - 用户说 SPEC、PRD、需求文档、feature brief 时，输出 SPEC。
   - 如果面向主管、同事、业务方发送，再用 `business-writing-coach` 改成真实工作表达。

## 成熟度判断

先判断当前处在哪一层，再行动：

| 层级 | 判断标准 | 下一步 |
|---|---|---|
| L0 点子 | 只有“想做一个 X” | `idea-exploration` |
| L1 需求雏形 | 有场景，但用户/数据/边界不全 | 产品化扩展 + 一问一答 |
| L2 方案假设 | 有方案，但取舍、风险、非目标不稳 | grill-me 式压力追问 |
| L3 SPEC-ready | 真问题、用户、输入输出、P0、验收基本清楚 | 输出 SPEC |
| L4 执行-ready | 文件/系统/验收/验证路径明确 | `plan` / `ralplan` / 执行 |

## “聊透”的完成标准

至少拿到下面 7 项，才算可以输出 SPEC 或交给计划：

1. **真问题**：不是“做一个工具”，而是在什么场景下让什么更稳定、更快或更可控。
2. **使用者**：谁每天用，谁维护，谁拍板，谁受影响。
3. **输入输出**：数据从哪里来，最终给用户什么结果。
4. **规则口径**：系统怎么判断、排序、统计、异常处理或人工确认。
5. **非目标**：第一版明确不做什么，哪些诱人但暂缓。
6. **MVP 切分**：P0/P1/P2 和暂缓项清楚。
7. **验收方式**：怎么证明需求有价值，怎么判断第一版成功。

如果缺 1-2 项但不影响判断，可以输出 SPEC 并把它们放到“开放问题”。如果缺 3 项以上，继续追问，不要急着写文档。

## 追问规则

- 一轮只问一个问题。
- 问题必须落在当前最高风险缺口上。
- 每个问题都先给推荐答案。
- 能从现有 Wiki、仓库、样例数据、用户材料判断的，先查证，不把事实问题甩回用户。
- 不问“你还想补充什么”这种低杠杆问题。
- 不把 brainstorming 扩成无边界大方案；每次扩展后都要切回 P0。

推荐格式：

```markdown
我建议先定这个问题：...

我的推荐答案是：...

为什么：...

你只需要回答：同意这个方向，还是要改成 X？
```

## 压力追问清单

当进入 grill-me 式压力测试时，优先挑一个问：

- 这个需求如果不做，真实损失是什么？
- 第一版最小可验证场景是哪一个？为什么不是更大的版本？
- 这件事谁会每天使用，谁只是偶尔看结果？
- 哪个输入数据最不稳定？没有它还能不能跑 P0？
- 哪个判断必须人工确认，不能交给 AI 自动决定？
- 如果系统给错结果，谁兜底，怎么回滚？
- 第一版明确不做哪些功能，防止需求膨胀？
- 3 个真实案例能不能覆盖 80% 的规则？
- 验收指标是节省时间、减少错误、提高响应速度，还是让流程可追踪？

## SPEC 模板使用指南

### 页面交互的写法与顺序

页面交互必须按**入口 → 展示内容 → 操作项及点击效果 → 特殊状态文案**这个顺序写，用无序列表，不合并不跳过。

**顺序的逻辑**：这就是用户进入一个页面的实际体验路径——先找到入口，进去后看到什么内容，然后做操作，操作出异常时看什么提示。研发按这个顺序读 PRD，不需要来回翻找。

**❌ 错误写法**（把所有交互混在一段话里）：
> 用户可以在任务详情页查看推荐方案，支持手动调整组长和一线人员，调整后系统重新计算排期。

**✅ 正确写法**（按顺序分开）：
- **页面入口**：任务中心 → 点击任务名 → 进入任务详情页 → 「推荐方案」Tab
- **展示内容**：组长推荐卡片（Top3）含姓名/成熟度/历史通过率/推荐分解释；一线同学列表含层级/CPH/准确率/当前状态；排期预览含预计交付时间和交付可达性标记
- **操作项及点击效果**：点击「手动调整组长」→ 弹出组长选择器 → 选完后必须选改判原因标签 → 点「确认替换」→ 排期预览实时重算
- **特殊状态提示文案**：候选池为空时：”当前条件下无可用人员。建议：①检查技能标签是否过窄；②扩大业务线范围”

### 何时展开”功能需求”章节

当需求涉及多角色流程、多步骤配置、或多类型差异化行为时，必须展开第 5 章（按节点拆开）。判断标准：

- 流程有 ≥3 个节点（如 标注→质检→验收→错题）→ 按节点展开
- 同一功能在不同组件/类型上行为不同 → 用”按类型差异化行为”表格
- 有可配置参数 → 用”基础配置”表格，标清取值范围和约束

### 写好功能需求的 7 条铁律

1. **配置项必须有约束**：不是”支持配置人数”，而是”3≤M≤10，创建后不可修改”
2. **规则必须有公式或阈值**：不是”取高票结果”，而是”票数 > M/2 则采纳”
3. **差异化行为必须有举例**：不是”单选取高票”，而是”甲选A、乙选A、丙选B、丁选B → 结果：A（2票=2票时置空）”
4. **交互文案必须是原文**：不是”弹窗提示用户”，而是精确到弹窗文案内容
5. **边界条件必须有处理**：无结果时置空还是报错？创建后能不能改？取消时弹什么提示？
6. **点击效果必须写清楚**：不是”支持查看详情”，而是”点击查看详情 → 展开侧边栏，回显最后一次操作人的结果，默认展开作业记录面板”
7. **功能联动必须写明白**：不是”两个功能有关联”，而是”选择多人投票模式后自动勾选质检节点；取消质检时弹窗告知'不一致数据将无质检最终结果'”

### 验收条件的质量标准

每个功能节点的验收条件必须是**可以直接操作来确认**的，不能是模糊描述。

**❌ 不合格（无法验证）：**
- “系统正常运行”
- “用户可以看到结果”
- “功能工作正常”

**✅ 合格（可以点击确认）：**
- “点击「确认方案」→ 弹出二次确认弹窗，包含文案'确认后将锁定排期方案'”
- “主管确认后，任务列表中该任务状态从「待审核」变为「已确认」”
- “选择3个人员后，排期预览中的'预计完成时间'自动重新计算”

**判断标准**：把验收条件给一个没看过需求的人，他能不能独立点一遍然后告诉你”通过了”还是”没通过”。能 → 合格；不能 → 重写。

### “现有系统盘点”章节的作用

写新需求前先盘点参考系统（竞品、内部已有工具、前代版本），是防止遗漏功能点的最有效手段。盘点时按流程模块对齐，而不是按页面对齐——这样能发现”这个环节竞品有但我们没想到”的盲区。

## 从会议/讨论记录中提炼 MRD

当用户提供会议纪要、听记转写、讨论记录等材料时，不要直接输出 SPEC。先做一轮结构化提炼，再进入正常聊需求流程。

### 提炼步骤

1. **识别发言人角色**：谁是决策者（上级/评审者）、谁是执行方（产品/研发）、谁是业务方。不同角色的发言权重不同——决策者的质疑 > 执行方的描述 > 业务方的诉求。
2. **提取决策与共识**：标记”已确认””达成一致””定了”的内容。这些是 MRD 的硬约束，不可回滚。
3. **提取质疑与挑战**：决策者反复追问的点 = 当前方案的最大风险缺口。每个质疑对应一个待回答的开放问题或设计缺陷。
4. **提取行动项**：谁负责什么、什么时候完成。这些是后续跟进的依据。
5. **区分事实 vs 假设**：会议中说的”我们已经有XX数据”是事实还是需要验证？”应该可以接API”是确认过还是猜测？
6. **标注优先级信号**：决策者说”一期先做””这个放二期””P0必须有”等表述，直接映射到 MVP 切分。

### 提炼输出格式

```markdown
# <项目名> MRD 提炼

## 会议基本信息
- 时间 / 参与人 / 时长

## 已确认决策（不可回滚）
1. ...
2. ...

## 关键质疑与待回答
| 质疑内容 | 提出人 | 当前状态 | 影响范围 |
|---|---|---|---|
| ... | ... | 未回答/已有方向/需验证 | ... |

## 流程缺口（说了但没设计进去的）
1. ...

## 行动项
| 事项 | 负责人 | 截止时间 |
|---|---|---|

## 对现有 SPEC/PRD 的影响
- 需要新增的节点/功能：
- 需要修改的设计：
- 需要删除或暂缓的功能：
```

### 提炼后怎么衔接

- 如果提炼结果覆盖了”聊透7项”中的 ≥5 项 → 直接进入 L3 输出 SPEC，把未覆盖的放入开放问题。
- 如果关键质疑中有 ≥2 个未回答 → 先用 grill-me 式追问把这些问题压给用户确认，再写 SPEC。
- 如果流程和现有 SPEC 差异大 → 先更新 SPEC 的节点结构，再补细节。

### 提炼铁律

- **不要把会议里的”讨论了”等同于”决定了”**。只有明确表态确认的才算决策。
- **不要忽略沉默**。某个关键环节没人提 = 可能都没想过，不是不需要。
- **质疑比共识更有价值**。共识只告诉你做什么，质疑告诉你哪里会出问题。
- **行动项必须落到人**。没有负责人的行动项 = 不会发生的事。

## 输出策略

- 用户只是想继续聊：输出当前判断 + 下一问。
- 用户要”聊透”：持续推进成熟度，直到达到 L3 或用户选择停止。
- 用户要”生成文档”：输出 SPEC；缺口写入开放问题，不要编造事实。
- 用户要”拿去发”：用 `business-writing-coach` 改成对应受众能接收的表达。
- 用户要”开始做”：先确认 L4 是否满足；不满足则走 `plan` / `ralplan`。
- 用户给会议/讨论材料：先做 MRD 提炼，再按成熟度推进。

## 最短输出形态

普通对话优先短，不要每次都完整套模板：

```markdown
**当前判断**
...

**还缺的关键边界**
...

**我的推荐**
...

**下一问**
...
```
